# AGENTS.md

Guidance for future AI/code agents working in this repository.

## Mission

Maintain and evolve a Kotlin rating/simulation engine plus a Vue frontend that reads generated artifacts. Prefer small, targeted changes and verify both backend and frontend flows after edits.

## First 5 Minutes

1. Check repo state:
   - `git status --short`
2. Confirm toolchain assumptions:
   - Java 25 (sdkman-managed in this team)
   - Node 20 (`.nvmrc` in root and `web/`)
3. Run baseline checks if your change is non-trivial:
   - `./gradlew clean test`
   - `cd web && npm run build`

## Critical Invariants

1. Backend and frontend season rules must stay aligned.
   - If you change promotion/relegation or tie-break logic in `SeasonPredicter.kt`, update `web/src/simulator/SeasonSimulator.ts` (and likely `web/src/simulator/PoulePredicter.ts`).

2. Many files are generated and intentionally versioned.
   - Expect diffs in:
     - `web/public/ranking.csv`
     - `web/public/graph.csv`
     - `web/public/meta.json`
     - `web/public/*.json`
     - `web/public/csv/**`
     - `web/src/origins.ts`
     - `matches/current.txt`
     - `aggr/*.csv`
     - `tiers.csv`

3. Java 25 is required by build and CI.
   - See `build.gradle.kts` and `.github/workflows/pr.yml`.

4. `SeasonPredicter` CLI currently only consumes `args[0]` as date.
   - Existing workflow passes an extra arg (`zaal`) that is ignored.

5. Files/code blocks generated by AI agents should be annotated as such.

## Safe Editing Rules

- Do not hand-edit generated artifacts unless the task explicitly asks for it.
- Avoid broad formatting changes across large CSV/JSON/TXT datasets.
- Keep changes localized to relevant modules; this repo has many large files and pre-existing historical data.
- `web/dist` and `web/node_modules` are build outputs; do not commit them.

## Validation Matrix

For backend logic changes:
- `./gradlew clean test`
- `./gradlew run` (if output-producing behavior was changed)

For season simulation/export changes:
- `./gradlew predict --args="2026-01-01"`
- Spot-check generated files in `web/public/csv/` and `web/public/*.json`

For frontend/store/simulator changes:
- `cd web && npm run build`
- If routing/UI changed: `cd web && npm run dev` and manually sanity-check affected pages

## Data + Scraper Notes

- Scraper caches API responses in `cache/` using hashed request keys.
- `Scraper.forceNetwork` is `false` by default; avoid turning it on unless needed.
- Running the backend may update `matches/current.txt` and season special match JSON files.

## Tests and Confidence

- Unit test coverage is light.
- `src/test/kotlin/nl/korfbalelo/elo/GlickoTest.kt` is disabled.
- Prefer validating behavior via targeted simulation runs plus frontend build/type-check.

## When Unsure

- Read the closest Kotlin + TS counterpart before changing rules.
- Preserve existing Dutch domain terminology in UI/routes/data names.
- Report exactly which generated outputs changed and why.
